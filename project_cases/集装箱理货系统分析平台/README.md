# 产品四阶段流水线 — 使用说明

## 快速开始

### 第一步：复制项目模板

```
将整个 _project_template 文件夹复制到 project_cases/ 下，重命名为你的项目名。
```

### 第二步：打开 Claude Code 会话

在项目目录下打开 Claude Code。AI 启动后自动执行：

1. **定位项目根目录**：检测到 CLAUDE.md → 记录项目根路径
2. **读取流水线状态**：发现全阶段 `not_started` → 判断为新项目
3. **询问下一步**：
   > "新项目 {项目名}。建议从 Stage 1 调研开始。需要创建 /branch Master 吗？"

### 第三步：按阶段推进

```
Stage 1: 调研与资料收集（Master 会话）
  ↓ /branch 产品设计
Stage 2+3: 功能分析 + 线框PRD（产品设计 会话）
  ↓ /branch 原型开发
Stage 4: 高保真原型（原型开发 会话）
```

## 各阶段详细说明

### Stage 1 — 调研与资料收集

| 项目 | 说明 |
|------|------|
| **负责会话** | Master |
| **技能** | `competitive-research` |
| **启动方式** | 打开会话后说"开始竞品调研" |
| **产物** | `01-调研与资料/竞品调研报告.md` |
| **收尾** | AI 提示"可以 /branch 产品设计 进入下一阶段" |

**操作流程**：
1. AI 询问产品方向、目标行业、客群 → 你描述想法
2. AI 搜索竞品（直接竞品 3-5 + 泛行业标杆 3-5）
3. AI 构建功能对比矩阵 → 你确认
4. AI 输出竞品调研报告
5. 状态文件自动更新：stage-1 = done

### Stage 2+3 — 功能分析 + 线框PRD

| 项目 | 说明 |
|------|------|
| **负责会话** | 产品设计（Branch A） |
| **技能** | `prd-generator` |
| **启动方式** | 在终端输入 `/branch 产品设计` |
| **产物** | `02-产品设计/PRD.md`、`页面清单.yaml`、`线框图/`、`_同步校验记录.md` |
| **收尾** | AI 提示"可以 /branch 原型开发 进入下一阶段" |

**操作流程**：

**前半段（Stage 2 — 对话确认）**：
1. AI 精读调研报告
2. AI 引导你确认：产品定位 → 用户画像 → 核心场景 → 约束条件
3. AI 逐模块做功能决策（保留/增强/新增/删除）→ 你逐项确认
4. AI 梳理核心业务流程（Mermaid 图）→ 你确认
5. 所有确认后的内容暂存于对话，不创建独立文件

**后半段（Stage 3 — 文档产出）**：
1. AI 输出页面清单 (`页面清单.yaml`) → **你二次确认**（后续不再轻易改动）
2. AI 确认 DESIGN.md 设计基底 → 你确认或调整
3. AI 逐页画线框图（Excalidraw）→ 每页你确认后再画下一页
4. AI 基于线框图生成完整 PRD.md（8 章 + 附录）
5. AI 执行线框图 ↔ PRD 同步校验 → 输出 `_同步校验记录.md`
6. 状态文件自动更新：stage-2 = done, stage-3 = done

### Stage 4 — 高保真原型

| 项目 | 说明 |
|------|------|
| **负责会话** | 原型开发（Branch B） |
| **技能** | `make-prototype-generator` |
| **启动方式** | 在终端输入 `/branch 原型开发` |
| **前置条件** | 先在 make 项目目录下新建好项目文件夹 |
| **产物** | 原型代码 + `.spec/checklist.md` + `acceptance-checklist.md` + `acceptance-report.md` |
| **收尾** | 全流程完成 |

**操作流程**：

1. AI 询问 make 项目路径 → 你提供绝对路径
2. AI 读上游产物：PRD.md + 页面清单.yaml + 线框图 + DESIGN.md
3. AI 输出**开发核对清单** + **验收检查清单** → 你确认
4. AI 选覆盖面最广的页面作为**样板页面** → 完整开发 → **你目视确认**
5. AI 按样板标准**批量产出**其余页面
6. AI 执行验收：编译验收 + 逐条业务验收 + 客户端演示检查 → 输出验收报告
7. 状态文件自动更新：stage-4 = done

## 迭代与反馈

### 上游改了 → 下游更新（顺流迭代）

1. 切到下游会话
2. 说"上游 XXX 改了，更新下游产物"
3. AI 自动读 pipeline-status.yaml → 对比版本号 → 判定影响范围 → 执行更新

### 原型发现设计问题 → 回溯上游（逆向迭代）

1. 原型开发会话中，AI 草拟反馈内容 → **你审核确认** → AI 写入 `_待反馈问题.md`
2. 切回产品设计会话 → AI 启动时自动检测待处理反馈 → 逐条处理
3. 切回原型开发 → 顺流更新验证

### 只改设计样式

1. 直接修改 DESIGN.md 文件
2. 切到原型开发会话，说"DESIGN.md 更新了，刷新视觉"
3. AI 重读 DESIGN.md → 更新 Token → 不涉及功能

## 关键文件说明

| 文件 | 作用 | 谁写 |
|------|------|------|
| `CLAUDE.md` | AI 操作指令集。AI 启动后第一个读的文件 | 模板自带 |
| `DESIGN.md` | 设计基底。定义颜色/字体/间距/组件/页面模板 | 模板自带（v3.1） |
| `pipeline-status.yaml` | 全局状态文件。记录所有阶段的进度和版本 | AI 自动维护 |
| `_待反馈问题.md` | 跨分支反馈载体 | AI 起草，你审核后写入 |
| `_迭代日志/` | 每次迭代的详细记录 | AI 自动追加 |

## 注意事项

- **所有 `/branch` 命令由你在终端输入**，AI 不会自动执行
- **每阶段结束时 AI 会提示你下一步**，你决定是否继续
- **管线状态文件不要手动编辑**，由 AI 按规范维护
- **页面清单确认后不要轻易改动**，会影响原型范围
- **DESIGN.md 是项目级设计标准**，所有原型严格遵循
