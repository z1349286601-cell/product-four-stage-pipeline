## 二、axhub make 本质：不只是原型工具

### 2.1 核心理念：「原型即 PRD」

```
传统模式：                       make 模式：
PRD 文档（主）                   可交互原型 + 标注（主）
    +                               +
原型（辅助）                      PRD 文档（补充判断依据）
```

### 2.2 三大核心能力

#### 能力一：标注系统（`@axhub/annotation`）

这是 make 最核心的差异化能力。在可交互原型上**任意元素**添加标注：

**标注数据结构**（annotation-source.json）：
```json
{
  "id": "唯一标识",
  "title": "标注标题",
  "pageId": "所属页面",
  "locator": { "selectors": [".css-selector"] },
  "aiPrompt": "AI 生成标注内容的提示词",
  "annotationText": "简短文字说明",
  "hasMarkdown": true,          // 是否含富文本
  "color": "#D97706",           // 颜色分类
  "controls": [{                // 状态切换控件
    "type": "segmented",
    "attributeId": "result_state",
    "options": ["成功", "失败"]
  }]
}
```

**标注可以做**：
- 任意元素的内容说明（按钮、卡片、表格、标题）
- 状态标注（成功/失败、空/有数据、指标偏低/正常/偏高）
- 同一元素多颜色分层标注（内容说明 + 分类 + 筛选）
- 颜色筛选（评审时只看某一类标注）

#### 能力二：原型目录系统

```
原型目录
├── 页面（切换 route，进入真实页面上下文）
├── 文档（README、PRD分章节、规则说明、验收清单）
└── 外部链接（设计来源、竞品参考）
```

**关键价值**：原型 + 标注 + 文档 + 外部参考，**一个入口全部管理**。

#### 能力三：内置 Review 体系

| Review 类型 | 关注点 | 输出位置 |
|------------|--------|---------|
| 原型 Review | 业务完整性、逻辑连贯性、数据一致性 | `.spec/prototype-review.md` |
| UI Review | 视觉美观、品牌一致性、响应式 | `.spec/ui-review.md` |

## 三、make 对现有流程的冲击分析

### 3.1 范式转变

```
现有流程：
  调研 → PRD → 原型 → 迭代（原型改了PRD没同步）⚠️ 分叉

引入 make 后：
  调研 → PRD → 原型 + 标注 → 迭代（标注即文档，不分离）✅ 统一
```

### 3.2 最大价值点

你提到的几个吸引力，在 make 里是这样实现的：

| 你的需求 | make 的实现方式 |
|---------|---------------|
| 在原型上随意批注 | annotation-source.json + `@axhub/annotation` 运行时 |
| 快速复制提示词 | 每个标注节点自带 `aiPrompt` 字段 |
| PRD + 原型放到一起管理 | 原型目录：页面 + docs/*.md + 外部链接 |
| 后台逻辑/隐藏说明标注到页面 | Markdown 标注 + 多颜色分层 |
| 在页面上生成完整 PRD | docs/ 目录直接挂 Markdown，可 AI 生成 |
| 传达给开发的信息统一管理 | 原型源码 + 标注 + 文档 = 统一的交付上下文 |

### 3.3 潜在新问题（你的疑虑）

| 风险点 | 分析 |
|-------|------|
| **技术栈绑定** | make 强制 React 18 + Ant Design 5 + Tailwind，与你传统方案的 Vue 3 不兼容 |
| **学习成本** | annotation-source.json 结构 + rules/ 工作规则 + DESIGN.md 主题系统，理解门槛不低 |
| **流程侵入性** | AGENTS.md 定义了严格的门禁（需求确认→设计确认→实现），可能与你现有节奏冲突 |
| **双轨维护** | 如果传统方案和 make 方案并存，两套技术栈+两套流程的维护成本 |
| **调研→make 断层** | competitive-research 输出调研报告.md，make 需要的是 `src/resources/` 下的结构化资料 |
| **PRD→make 断层** | prd-generator 输出完整 PRD.md（8章），make 的 docs/ 期望按主题拆分的轻量 markdown |
| **反向同步未验证** | make 解决「原型→标注」的正向，但「标注→PRD 文档」的自动反向生成能力待验证 |

## 四、阶段三→阶段四 衔接分析

### 当前两条路径

```
路径A：传统方案
  PRD.md → 传统提示词（Vue3）→ 独立工程原型 → 人工对照PRD修改

路径B：make 方案
  PRD.md → make提示词（React）→ @axhub/make-client 内原型 + 标注 → 标注迭代
```

### 衔接断点

| 维度 | 路径A | 路径B |
|------|-------|-------|
| 原型→PRD同步 | 无机制 | annotation 可生成 markdown 文档，但需验证 |
| PRD→原型追溯 | 靠命名约定 | annotation-source.json 的 pageId 可追溯 |
| 变更管理 | 散落 | .spec/ 目录归档决策快照 |
| 多方评审 | 无内置 | prototype-review + ui-review |
