> 对应流水线 v1.0 | 最后更新：2026-07-03
> 全阶段通用 | 依赖：pipeline-status.yaml + _待反馈问题.md + _迭代日志/ + _流程问题经验.md
> 详细规则见各项目 CLAUDE.md 第六章，本文档为人类阅读的方法论参考。

## 三种迭代模式

### 1. 顺流迭代（上游改 → 下游更新）

```
触发：上游 Branch 修改产物 → version++ → downstream_impact 中列出的 stage.status = needs_update
检测：下游 Branch 启动必检 → 发现 status = needs_update → 提示用户
执行：用户确认 → 读新上游文件 → 根据 impact_type 判定范围 → 执行更新 → 状态恢复 done
```

**影响范围判定**（基于 pipeline-status.yaml 的 `downstream_impact.impact_type`）：

| 类型 | AI 行为 |
|------|--------|
| 文案级 | 搜索替换对应文案 |
| 布局级 | 调整组件位置/尺寸 |
| 功能级 | 新增/删除/修改页面组件和交互 |

### 2. 逆向迭代（下游发现问题 → 回溯上游）

```
触发：原型开发（Branch B）发现 PRD/线框图设计问题

操作路径：
  1. Branch B 草拟反馈内容 → 提交用户审核
  2. 用户确认后 → Branch B 写入 _待反馈问题.md（状态=待处理）
  3. /resume 产品设计
  4. Branch A 启动必检 → 发现待处理反馈 → 逐条处理
  5. Branch A 修改上游产物 → 标记反馈状态=已处理 → version++
  6. /resume 原型开发
  7. Branch B 顺流更新 → 验证修复效果
```

### 3. 最小影响迭代（只改 DESIGN.md）

```
触发：用户直接改 DESIGN.md 文件
操作：/resume 原型开发 → 重读 DESIGN.md → 更新 Token → 目视确认
影响：不涉及功能清单/线框图/PRD
```

## 发起位置矩阵

| 从哪发起 | 改了什么 | 需要更新 | 路径 |
|---------|---------|---------|------|
| Master | 调研新增竞品 | 全链路 | 产品设计→原型开发 |
| 产品设计 | 功能清单增删 | Stage 3 + Stage 4 | 产品设计→原型开发 |
| 产品设计 | 线框图调整 | Stage 4 | 原型开发直接更新 |
| 产品设计 | PRD 文案修改 | Stage 4 | 原型开发直接更新 |
| 原型开发 | 原型样式微调 | 无 | 不回溯源 |
| 原型开发 | 发现设计不合理 | Stage 3 | 写反馈→产品设计→回原型开发 |
| 全局 | DESIGN.md | Stage 4 | 原型开发刷新视觉 |

## _待反馈问题.md 格式

```markdown
### FB-XXX
- **发现时间**：YYYY-MM-DD
- **来源 Branch**：原型开发
- **问题页面**：PC-XX
- **问题描述**：...
- **建议方案**：...
- **状态**：待处理 | 已处理 | 已拒绝（附原因）
```

⚠️ **审核门**：Branch B 草拟反馈 → 用户确认 → 才落笔写入文件。不允许 AI 直接写入。

## 双轨迭代记录

| 存储位置 | 内容 |
|---------|------|
| `pipeline-status.yaml → iterations[]` | 元数据索引（id/date/trigger_stage/status） |
| `_迭代日志/<id>.md` | 完整详情（触发原因/改动文件/下游更新/验证结果） |

## needs_update 流转规则

```
触发：上游 version++ → 被 downstream_impact 引用的 stage.status = needs_update
检测：下游 Branch 启动必检（CLAUDE.md 第零章步骤 4）
清除：顺流更新完成 → status 恢复 done → synced_upstream_versions 刷新
禁止：手动把 needs_update 改回 done（必须先执行顺流更新）
```

## 状态回退规则

```
正向：not_started → in_progress → done
回退：
  in_progress → not_started ：允许
  done → not_started ：禁止（走迭代流程）
  done → needs_update ：允许（仅上游变更触发）
  needs_update → done ：允许（顺流更新完成后）
```

## 并发写入防护

修改 pipeline-status.yaml 前必须：
1. 二次读取文件
2. 对比 `pipeline.updated` 时间戳
3. 不一致 → 暂停，提示用户可能有其他 Branch 同时在写
