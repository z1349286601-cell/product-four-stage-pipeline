## 三、启动机制

### 3.1 新项目启动（双模板）

```
Step 1: 复制流水线项目模板
  cp -r _project_template/ project_cases/<新项目名>/

Step 2: 复制 make 工程模板（手动操作）
  cp -r make-project-template/ D:/AI/workspace/axhub_workspace/<新make项目>/

Step 3: 安装依赖（手动操作）
  cd <新make项目> && npm install

Step 4: 打开 Claude Code 会话
  AI 自动执行启动必检（见 CLAUDE.md 第零章）

Step 5: 按阶段推进
  Stage 1 → /branch 产品设计 → Stage 2+3 → /branch 原型开发 → Stage 4
```

### 3.2 AI 启动必检（CLAUDE.md 第零章）

每次会话打开后 AI 自动执行：

1. **定位项目根目录** — CLAUDE.md 所在目录 = `{PROJECT_ROOT}`
2. **读取流水线状态** — pipeline-status.yaml，自动填入 project_root
3. **意图分析 → 阶段判定** — 基于用户操作意图（非会话名关键字）映射到阶段
4. **前置条件检查** — 下游阶段的前置上游必须 done
5. **当前状态检查** — needs_update / in_progress / done 的分支处理
6. **上游文件校验** — 检查上游文件存在性
7. **版本对比** — synced_upstream_versions vs 当前版本，检测上游变更
8. **输出路径速查** — 项目根、make 项目、上游产物三条路径

## 四、三技能驱动机制

流水线的执行逻辑不再写在 CLAUDE.md 的冗长流程中，而是由三个 Skill 文件承载。CLAUDE.md 负责"会话启动时做什么"，Skill 负责"进入阶段后怎么做"。

| Skill | 阶段 | 核心原则 |
|-------|:--:|---------|
| `competitive-research` | Stage 1 | 结构化澄清对话 → 七步调研法 → 单一报告输出 |
| `prd-generator` | Stage 2+3 | 大纲驱动 → 分两段执行 → 线框图权威源 → 逐页确认 → 单一 PRD.md |
| `make-prototype-generator` | Stage 4 | 模板启动（非从零手写）→ 样板确认 → 批量产出 → 合规自检 → 验收修复 |

### 4.1 Skill 的触发

- 用户在对话中提到对应意图（如"生成原型"、"做PRD"、"竞品调研"）
- AI 识别意图后调用对应的 Skill 工具
- Skill 文件加载后，AI 按 Skill 中定义的步骤执行

### 4.2 Skill 与 CLAUDE.md 的分工

| 维度 | CLAUDE.md | Skill 文件 |
|------|-----------|-----------|
| 读取时机 | 每次会话启动 | 用户触发对应意图时 |
| 作用范围 | 所有阶段通用（启动、迭代、状态更新） | 单个阶段的执行细节 |
| 内容 | 第零章启动必检、迭代规则、状态文件规范 | 步骤流程、技术栈、布局约束、验收标准 |

## 五、流水线状态协议（pipeline-status.yaml v2.2）

所有会话共享读写的中央状态文件，是流水线的"神经系统"。

### 5.1 结构

```yaml
pipeline:
  project: "产品全称"
  project_root: "CLAUDE.md 所在目录的绝对路径"  # AI 首次读取自动填入
  make_project:
    path: "make 项目绝对路径"        # Stage 4 启动时用户提供
    theme: "src/themes/antd-pro/DESIGN.md"
    status: "pending | ready"

stages:
  - id: stage-1~4
    status: "not_started | in_progress | done | needs_update"
    outputs[]:                       # 每阶段产出文件清单
      path: ...
      version: N
      downstream_impact[]:           # 文件级影响声明
        target_stage: ...
        impact_type: "文案级 | 布局级 | 功能级"
    synced_upstream_versions[]:      # 上游版本快照（用于变更检测）

iterations: []                       # 迭代元数据索引
```

### 5.2 needs_update 流转

```
上游修改产物 → version++ → 下游 stage.status = needs_update
  → 下游启动必检时发现 → 提示用户执行顺流更新
  → 顺流更新完成 → status 改回 done
```

### 5.3 写入防护

每次修改 pipeline-status.yaml 前，AI 必须二次读取并对比 `pipeline.updated` 时间戳，防止并发写入冲突。

## 六、迭代机制

### 6.1 三种迭代路径

| 路径 | 触发 | 流程 |
|------|------|------|
| **顺流** | 上游改 → 下游需更新 | 下游启动必检发现 needs_update → 执行更新 |
| **逆向** | 原型发现设计问题 | 草拟反馈 → 用户审核 → 写入 `_待反馈问题.md` → 产品设计处理 → 顺流更新 |
| **最小影响** | 只改 DESIGN.md | 原型开发直接刷新 token，不回溯功能分析 |

### 6.2 反馈闭环（`_待反馈问题.md`）

- AI 草拟反馈内容 → **用户审核确认** → AI 写入文件
- 产品设计会话启动时自动检测待处理反馈
- 状态：待处理 / 已处理 / 已拒绝

### 6.3 双轨迭代记录

- `pipeline-status.yaml → iterations[]`：元数据索引（id/date/trigger_stage/status）
- `_迭代日志/<id>.md`：完整详情（触发原因、改动文件、验证结果）

## 七、流程问题经验反馈闭环

每次全流程实操中发现的问题，记录到项目的 `_流程问题经验.md`，经分析后回灌到流程资产中：

```
项目实操 → 发现问题 → _流程问题经验.md → 分析归类 →
  ├─ Stage 3 问题 → 回灌 prd-generator skill + CLAUDE.md 第四章
  ├─ Stage 4 问题 → 回灌 make-prototype-generator skill + CLAUDE.md 第五章
  ├─ 通用问题 → 回灌 CLAUDE.md 第零章
  └─ 模板问题 → 回灌 _project_template/ 或 make-project-template/
```

最近一轮（集装箱理货系统分析平台，2026-07-03）发现 12 条问题，已全部回灌：

| 来源阶段 | 问题数 | 回灌位置 |
|:--:|:--:|------|
| Stage 3 | 3 | CLAUDE.md 4.4.1~4.4.3 + prd-generator skill 3.0.1 |
| Stage 4 | 9 | CLAUDE.md 5.1.5/5.5.5/5.5.6 + make-prototype-generator skill + 合规清单 + 流程检查点 |
