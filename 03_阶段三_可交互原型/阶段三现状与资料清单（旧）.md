## 二、两套方案对比分析

### 2.1 传统从零开发方案

```
执行模式：全程自治，单段 prompt 驱动，不等待用户确认
输出物：独立 Vue3 工程，无需依赖特定项目框架
```

| 维度 | 详情 |
|------|------|
| **技术栈** | Vue 3 + Vite + TypeScript (ESM) |
| **UI 框架** | Ant Design Vue 4.x |
| **图表** | vue-echarts + ECharts 5.x |
| **路由** | vue-router 4.x (SPA) |
| **地图** | 高德地图 JS API |
| **图标** | @ant-design/icons-vue |
| **样式** | Scoped CSS + 全局 UI 常量文件 |
| **数据** | Mock 数据，api/ 目录封装 |

**五阶段执行流程**：
```
阶段1：工程初始化与基建（package.json / 目录 / 路由 / Mock）
阶段2：公共组件与图表封装（通用表格/表单弹窗/筛选栏/ECharts容器）
阶段3：核心页面逐页开发（按PRD第6章逐页开发+自检）
阶段4：全量交互联调（跳转/弹窗/校验/分页/排序/异常态）
阶段5：全量验收与闭环修复
```

**五级验收体系**：
| 校验级 | 内容 | 判定 |
|:---:|------|------|
| 校验0 | 业务一致性（页面匹配对照表） | 逐页对标 |
| 校验1 | 编译运行（0 error, 0 warning） | npm run build |
| 校验2 | UI 还原度（色值/字号/间距/状态） | 逐项比对 |
| 校验3 | 交互功能（搜索/表格/弹窗/图表/反馈） | 逐功能测试 |
| 校验4 | 代码规范（TS类型/常量/命名/目录） | 静态检查 |

**特点**：
- ✅ 单段 prompt，全程自治，不打断用户
- ✅ 五级验收，质量有保障
- ✅ 技术栈独立，不依赖特定项目框架
- ❌ 全程黑盒，中间结果不可见
- ❌ 发现问题只能等最终验收报告
- ❌ 不适合与 make 版本的项目体系衔接

### 2.2 make 版本方案（axhub make）

```
执行模式：4 阶段递进，每阶段等待用户确认
输出物：@axhub/make-client 项目体系内的原型
```

| 维度 | 详情 |
|------|------|
| **技术栈** | React 18 + TypeScript (Classic JSX) |
| **UI 框架** | Ant Design 5.x + ConfigProvider |
| **样式** | Tailwind CSS v4 + Design Token |
| **图表** | echarts 6.x + echarts-for-react 3.x |
| **路由** | 单页：index.tsx 直接导出；多页：useHashPage |
| **构建** | Vite 5.4 MPA，入口自动注册 |
| **地图** | 高德地图 JS API（同 key） |

**四阶段执行流程**：
```
阶段1：需求读取与图表选型 → 输出核对清单 + DESIGN.md 确认
  🛑 等待用户确认

阶段2：样板页面开发 → 1个覆盖面最广的页面完整开发
  🛑 等待用户目视确认

阶段3：批量产出其余页面 → 复用样板，逐页开发
  🛑 等待用户目视验收

阶段4：验收与修复 → check-app-ready.mjs 脚本验收
  🛑 完成
```

**图表选型决策树**（make 版本特有）：
```
排名对比 → 横向柱状图 | 时序趋势 → 折线图 | 分组比较 → 纵向柱状图
占比构成 → 环形图 | 二维交叉 → 散点图 | 层级/流向 → 树图/桑基图
多指标画像 → 雷达图 | 单值状态 → 仪表盘
```

**验收标准**：
- `check-app-ready.mjs` 返回 `status: READY`
- 页面数量/名称/模块/字段与需求文档一致
- UI 与 DESIGN.md 一致，无硬编码色值
- 交互/状态/兜底齐全

**特点**：
- ✅ 分阶段确认，用户可控
- ✅ "样板确认→批量产出"模式，降低大面积返工风险
- ✅ 项目体系标准化（目录/脚本/规范内置）
- ✅ 图表选型决策树，减少设计反复
- ❌ 需要等待用户确认，不能全自动
- ❌ 依赖 @axhub/make-client 项目环境

### 2.3 两套方案对比矩阵

| 对比维度 | 传统从零开发 | make 版本 |
|---------|:-----------:|:--------:|
| **执行模式** | 全自动单段 | 4 阶段需确认 |
| **框架** | Vue 3 | React 18 |
| **组件库** | Ant Design Vue 4 | Ant Design 5 |
| **样式方案** | Scoped CSS + 常量文件 | Tailwind CSS v4 + Token |
| **构建** | Vite SPA | Vite MPA |
| **验收方式** | 五级手工校验 | 内置脚本 + 人工 |
| **项目依赖** | 无，独立工程 | @axhub/make-client |
| **中间可见性** | 低（黑盒） | 高（分阶段） |
| **设计系统** | 无内置 | DESIGN.md 主题库 |
| **用户参与度** | 低 | 中 |
| **适合场景** | 快速出整体原型 | 需精雕细琢的原型 |

## 三、衔接问题诊断

### 阶段二(PRD) → 阶段三(原型)

**两套方案共同的输入**：PRD「详细功能页面设计」章节

**传统方案对 PRD 的消费方式**：
```
PRD 全文 → 提取第6章全部页面清单 → 形成开发核对清单 → 逐页落地
```
- 依赖 prompt 中的「阶段 3：核心页面逐页开发」逐页自检
- 但没有结构化的「页面→代码」映射机制

**make 方案对 PRD 的消费方式**：
```
PRD 第6章 → 阶段1输出《开发核对清单》+ 图表选型标注 → 用户确认 → 开发
```
- 多了「图表选型」这个智能增强点
- 核对清单需要用户确认，多一道把关

**共同痛点**：PRD 不是为原型生成而写的，需要 AI 从文档中提取结构化信息

### 阶段三(原型) → 阶段四(确认迭代)

**传统方案**：验收报告 + 循环修复 → 但没有与 PRD 的回溯机制
**make 方案**：每阶段确认 + 验收脚本 → CLAUDE.md 提到「需求或设计变更要回到对应阶段对齐」
